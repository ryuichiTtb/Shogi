# カード将棋 設計ドキュメント

> **関連Issue**: [#68](https://github.com/ryuichiTtb/Shogi/issues/68)
> **ブランチ**: `feature/#68`
> **ステータス**: Phase 0(大枠の側)実装方針確定、本実装フェーズ着手中

本ドキュメントは、既存の将棋アプリに「カードゲーム要素」を追加するプロジェクトの設計記録。議論の進展に合わせて継続的に更新する。

---

## 1. プロジェクトゴール

既存の将棋アプリに、マナ管理・カードドロー・トラップ等のカードゲーム的要素を加えた**新モード**を追加する。従来将棋を楽しみたいユーザーと、新しい遊び方を試したいユーザーの両方に訴求する。

---

## 2. 大枠の決定事項

### 2.1 対戦形態(軸1)

| フェーズ | 対戦形態 | 備考 |
| --- | --- | --- |
| **MVP** | vs AI のみ | 既存の Claude SDK ベース AI を拡張 |
| **将来** | オンラインPvP を追加 | リアルタイム通信方式は別途検討 |

- **方針**: まずは vs AI で新ルールの面白さを検証し、その後オンライン化する段階導入。
- オンラインPvP は「確実に実施する」ため、MVP の設計段階からデータモデルや状態管理で**後続拡張を妨げない構造**を意識する。

### 2.2 モード構成(軸2)

- **従来将棋モード**(通常ルール / カード無し)
- **カード将棋モード**(マナ・カード・トラップ等の新ルール)
- **ホーム画面で切り替え可能**にする(トップページから明示的に選択)

### 2.3 マネタイズ(軸3)

- **現段階では保留**。MVP では課金要素を実装しない。
- 将来的なガチャ・スキン課金・買い切り等は、MVP リリース後の運営判断にて決定。

### 2.4 AI のカード運用(軸4)

- AI にもカードを使わせる(将来的に)。
- **既存 AI は Alpha-Beta 探索ヒューリスティック実装**(`src/lib/shogi/ai/engine.ts`、`search.ts`、`evaluate.ts` 等)。Claude SDK は現時点で未使用。
- **Phase 0 ではカード判定はせず、既存 `calculateAiMove` をそのまま流用**(card-shogi variant でも駒の指し手のみ、カード使用は実質パス)。
- **Phase A 以降で簡易ヒューリスティック評価関数を拡張**し、AI も手札・マナ・トラップを考慮した着手判断を行う。
- **将来的に Claude API 導入も検討事項**(コスト・レイテンシ要評価)。手札・マナ・盤面状況をプロンプトで渡して高度な判断をさせる構想。
- 難易度ごとに「使用するカードのレア度や判断基準」を変えることで、既存の難易度システムと親和させる想定。

### 2.5 MVP スコープ / リリース計画(軸5)

段階的に拡張していく方針。最終的には**フルMVP(C)まで必ず到達**させる。

| フェーズ | 内容 | 備考 |
| --- | --- | --- |
| **Phase 0(本Issue)** | 大枠の側 + 状態管理の枠 + 仮カード3枚 + DBスキーマ拡張 | UI骨格・データモデル・状態遷移の基盤を整える。AIはカード未使用で流用 |
| **Phase A(初回リリース)** | カード将棋の中核ルール / 固定デッキ / カード5〜8種程度 / AIは1難易度 | まず「遊べる」状態で公開 |
| **Phase B** | 全カード実装 / 難易度3段階 / 棋譜保存 / トラップ演出 / Framer Motion 演出導入 | 実プレイを踏まえて調整 |
| **Phase C(最終)** | デッキ編成画面 / アニメーション完全版 / バランス調整済み | フル機能完成 |

### 2.6 アーキテクチャ方針(軸6)

**ハイブリッド(イベント駆動設計)** を採用。

- ゲーム状態は当面クライアントで保持・判定するが、**状態遷移を「イベント」として抽象化**する。
  - 例: `moveEvent`(通常の着手) / `cardPlayEvent`(カード使用) / `trapTriggerEvent`(トラップ発動) 等
- こうすることで、将来オンラインPvPを導入する際に「イベントをサーバーに送り、サーバー権威で判定する」構造へスムーズに移行できる。
- MVP ではサーバー権威化までは踏み込まない(実装コスト圧縮)。

### 2.7 アカウント / データ永続化(軸7)

**MVP は「サーバー保存 + 匿名ID」。将来的に「ゲスト+任意ログイン」の両対応に拡張**。

| フェーズ | 方式 | 挙動 |
| --- | --- | --- |
| **MVP** | 匿名ID(cuid)でサーバー保存 | 既存の `User` モデル(cuid)を活用。端末ごとにIDを発行し、サーバーに所持カード・デッキ・進捗を保存。機種変更時は引き継げない(=リセット扱い)。 |
| **将来** | ゲスト + OAuth(Google等)任意ログイン | 匿名IDの `User` に OAuth 情報を紐付けて、端末間引き継ぎを可能にする。 |

- MVP では **ログイン機能は実装しない**(ユーザー参入障壁を下げる)。
- 課金導入フェーズで OAuth を追加する想定。

### 2.8 既存モードとの共通化レベル(軸9)

**画面分離・ロジック共通化** 方針。

- **共通化する部分**: 盤面描画 / 駒ロジック / AI呼び出し基盤 / 棋譜保存基盤 / キャラクター選択
- **分離する部分**: ゲーム状態(マナ・手札・トラップ等) / ゲーム画面UI / ホーム画面上のモード選択
- `use-shogi-game` フックは従来将棋のコアロジックとして維持し、カード将棋は別フック(例: `use-card-shogi-game`)で包むイメージ。

### 2.9 ローカライズ(軸12)

- **日本語のみ**で確定。i18n 基盤は導入しない。

### 2.10 アセット調達方針(軸13)

- **SE / BGM**: 開発者が無料素材サイトから調達。まずは「どの演出で何の音を鳴らすか」の**演出ポイント**を実装で仕込んでおき、後から音源を差し替えて検証できるようにする。
- **カードイラスト**: MVP では**仮イラスト / 簡易デザイン**で作成。後から正式版に差し替え可能な構造にする。

### 2.11 Phase 0 確定事項(本Issue)

Phase 0a で UI 3案のモック評価を実施し、以下を確定。

#### 採用UI(レスポンシブ・ハイブリッド)

| 環境 | 採用案 | レイアウト概要 |
| --- | --- | --- |
| **PC**(`>=768px` / Tailwind `md:`) | **A案: 上下分割** | 相手のカード要素(マナ/裏向き手札/山札/トラップ)を画面上端、自分のカード要素を画面下端に配置。盤面と持ち駒は中央。 |
| **モバイル**(`<768px`) | **B案: ボトムドロワー** | 相手側は上端細バーにマナ/手札枚数/山札/トラップ(裏)を集約。自分側は下端に「手札を開く」ボタン+マナ+山札+トラップを配置し、ボタンタップで手札 Sheet が下からスライドアップ。 |

切替は `md:` breakpoint の CSS クラス制御で対応(JS の動的判定は使わない=SSR との整合性を保つ)。状態管理(`use-card-shogi-game`)は両環境で共通。

#### 不採用案と理由

- **A案単独**: モバイル縦長で盤面が圧迫されすぎる(評価軸「盤面の視認性(モバイル縦)」で最低)
- **B案単独**: PC 環境で手札の即時参照性が落ちる。広い画面を活かせない
- **C案(オーバーレイ + 縦サイドカード)**: PC/モバイルでレイアウトが大きく異なるためメンテナンスコスト増。モバイル時はA案類似に collapse するため独自性が活きない

> ※ モバイルでの B案 の操作感が不十分と判断された場合、`md:hidden` を外して モバイルでも A案 ブロックを表示する変更で容易に統一可能。

#### その他の確定事項

| 項目 | 確定内容 |
| --- | --- |
| ホーム画面のモード選択 | 最上段に Tabs トグル(従来将棋 / カード将棋(BETA))を追加 |
| ルーティング | `/game/[id]` を維持し variant 内部分岐(URL一本化) |
| DBスキーマ | `Card` / `PlayerCardCollection` / `Deck` / `DeckEntry` テーブル新設、`Game.cardState Json?` 追加 |
| AI層 | 既存 `calculateAiMove` を流用(card-shogi でも駒指しのみ、カード未使用) |
| カード使用とターン消費 | **カード使用 = 1手相当**(ターン消費する)。1ターン中の駒指し+カード使用は禁止 |
| 早指しタイマー基準点 | 自分の番開始時点の `Date.now()` を `lastTurnStartedAt` に格納 |
| リロード時の `pendingCard` | 復元しない(キャンセル相当) |
| イベントログ | 全状態遷移を `GameEvent` として `eventLog` に追記。将来サーバ権威化への布石 |

## 3. 現在検討中の仕様(仮置き / 確定前)

> 以下はあくまで初期アイデア。バランス・実装性を踏まえ継続調整する。

### 3.1 マナシステム

- 一手指すごとに **1マナ** チャージ
- **1秒以内の早指し**で **2マナ** チャージ
- **5マナ** 消費で山札から1枚ドロー
- カード使用には各カード固有の必要マナを消費

**論点**:
- マナ上限の有無(例: 最大10マナ)
- 早指しマナボーナスは「じっくり考える」将棋の本質と葛藤する懸念あり → ランクモード/カジュアルで差を付ける案も
- マナの使途(ドロー / カード使用)の比重

### 3.2 手番の選択

自分の手番で、手札にカードがある場合、以下を選択できる:
- 通常の一手を指す
- 手札からカードを1枚使用する

### 3.3 カード種別

#### 通常カード(案)
| カード名 | 効果概要 | 想定コスト | バランス懸念 |
| --- | --- | --- | --- |
| 2歩指し | 2歩の禁手制約を一時的に解除 | 中 | 要検証 |
| マナUP | マナを追加獲得 | 低 | 無限ループ防止の制約必要 |
| 桂馬戻り | 盤上の桂馬を持ち駒に戻す | 中 | 対象範囲要定義 |
| 歩戻り | 盤上の歩を持ち駒に戻す | 低 | |
| 香戻り | 盤上の香車を持ち駒に戻す | 中 | |
| 2手指し | 連続で2手指せる(王手→詰み禁止) | 高 | **非常に強力**。追加制約検討(2手目は駒取り禁止 等) |
| チェンジ・ザ・ワールド | 盤面反転 | 高 | 終盤使用で必至が解ける懸念。仕様明文化必要 |
| 時間(自分)UP | 自分の持ち時間増加 | 低 | |
| 時間(相手)DOWN | 相手の持ち時間減少 | 中 | |
| トラップ破壊 | 相手のトラップを破壊 | 低 | |
| 持ち駒破壊 | 相手の持ち駒を破壊 | 高 | **強力**。全破壊/1枚指定/ランダム で強度が激変 |

#### トラップカード(案)
| カード名 | 効果概要 | 発動条件 |
| --- | --- | --- |
| 成り無効化 | 相手の成り駒宣言を無効化 | 相手が成り宣言時に自動発動 |
| 王手駒取り | 相手が王手した駒を取得 | 相手の王手時に自動発動 |

**トラップルール**:
- 自分の盤面に **1枚だけ** セット可能
- 既にセット済みでも新規セット可。既存は破棄される

### 3.4 デッキ・カード入手

- 将来的にガチャ・課金要素を検討(MVP では実装しない)
- MVP では固定デッキ or 全カード使用可 等で検証する案

### 3.5 Phase 0 暫定実装カード(3種)

Phase 0 では、状態遷移と効果適用フローを最小実装で網羅検証するため、以下の3種を実装する。Phase A 以降で 3.3 の全11種に拡張予定。

| カード | 種別 | コスト(マナ) | 効果 | 検証する側面 |
| --- | --- | --- | --- | --- |
| **マナUP** | 通常 | 2 | 即時マナ +3 | 状態のみ変化、ターゲット選択不要 → 最もシンプルなフロー |
| **歩戻し** | 通常 | 3 | 自盤上の歩1枚を持ち駒に戻す | ターゲット選択UIフロー(盤面ターゲット指定) |
| **成り無効化** | トラップ | 4 | 次の相手の成り宣言を1回無効化 | トラップセット → 自動発火フロー |

選定方針: 「状態のみ」「ターゲット選択あり」「トラップ」の3パターンを最小被覆することで、Phase A 以降のカード追加時に既存パイプラインがそのまま流用できることを担保する。

仮の数値(マナ上限 10、ドローコスト 5、初期マナ sente=3 / gote=1、初期手札 各2枚、固定デッキ 各6枚)は Phase 0 の動作確認用。Phase A でバランス調整。

## 4. アーキテクチャ方針(暫定)

### 4.1 技術スタック(既存継承)
- **Next.js 16 / React 19**
- **Prisma 7 + Neon (Postgres)**
- **Claude SDK**(現時点では未使用。将来 AI 拡張で導入検討)
- 既存 AI は **Alpha-Beta 探索ヒューリスティック**(`src/lib/shogi/ai/`)
- **Howler**(サウンド) / **Shadcn UI**(UI キット)
- **Framer Motion**(Phase B 以降で導入予定。Phase 0 / Phase A は CSS transition で対応、現時点未使用)

### 4.2 拡張予定箇所
- **ルーティング**: ホーム画面にモード選択 UI、カード将棋用ゲーム画面
- **ゲーム状態管理**: 既存 `use-shogi-game` を維持し、カード将棋は **別フック** (例: `use-card-shogi-game`) を新設して状態(マナ / 手札 / トラップ / 山札)を管理
- **データモデル**: カード定義 / プレイヤー所持 / ゲーム内カード使用履歴 の追加(Prisma スキーマ拡張)
- **AI 層**: MVP は簡易アルゴリズム実装(`lib/shogi/` 配下にヒューリスティック評価モジュール追加)。将来 Claude 呼び出しに切り替え可能な抽象化を行う
- **UI コンポーネント**: 手札表示 / 山札 / マナゲージ / トラップスロット / カード使用演出
- **サウンド演出フック**: 演出ポイントに Howler 呼び出しを仕込み、後から音源差し替え可能にする

### 4.3 将来のオンラインPvP への備え
- ゲーム状態遷移を **イベント駆動**(`moveEvent` / `cardPlayEvent` / `trapTriggerEvent` 等)で表現し、後のサーバー権威化・リアルタイム同期に転用しやすくする
- クライアントロジックとイベント適用ロジックを分離して実装する

### 4.4 後回しとした軸(大枠検討で保留)

以下は MVP 前後で判断するため保留。

- **軸10**(勝敗・終了条件の整理): カードで将棋の前提が崩れる場合の判定ルール
- **軸11**(プラットフォーム戦略): Web PWA 継続前提で進行。モバイルネイティブ化は将来判断
