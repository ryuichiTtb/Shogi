## 絶対ルール

### 1. PR作成・マージ・Issueクローズは明示的指示まで禁止

PR作成・マージ(`gh pr merge`)・Issueクローズ(`gh issue close`)は、ユーザーから明示的に指示があるまで**絶対に実行しない**。

**Why:** ユーザーは変更内容を自分で確認してからマージ・クローズしたい。事前確認なしの実行は信頼を損なう。

**How to apply:**
- コード変更が完了したらブランチをプッシュして止まる
- 「確認いただけましたら指示をお願いします」と明示的に伝える
- 「対応してください」「Issue#XXを対応してください」と言われても、PR・マージ・クローズは別途指示が来るまで実行しない
- **唯一の例外:** ユーザーが「PR・マージ・クローズまでお願いします」と一連の作業として明示的に依頼した場合

### 2. Issue対応時は必ずブランチを作成

Issue対応は必ず専用ブランチを作成してから実装を開始する。**mainブランチへの直接コミット・プッシュは禁止**。

**Why:** mainには動作検証済みのコードのみをマージしたい。

**How to apply:**
1. Issue対応開始前に `git branch -a` で既存ブランチを確認する
2. 対象Issueに関連しそうなブランチが存在する場合、そのブランチが当該Issue用かユーザーに必ず確認する
3. ユーザーが「Yes」と答えた場合のみそのブランチに切り替える
4. 既存ブランチがない場合は新規作成する(命名規則は次節)
5. mainへのマージは「ルール1」に従い、明示的指示があるまで実行しない

### 3. ブランチ命名規則

ブランチ名は以下の prefix を使用し、**`{prefix}/#{Issue番号}`** の形式とする。

| prefix | 用途 |
|---|---|
| `feature/` | 新機能追加・機能拡張(ユーザーから見て新しいことができる変更) |
| `fix/` | バグ修正(意図通りに動いていなかったものを直す) |
| `refactor/` | 振る舞いを変えないコード整理・責務分離・命名変更 |
| `chore/` | 上記3つに当てはまらない雑務(依存関係更新、設定変更、CI調整等) |

**例:**
- `feature/#69` — Issue #69 で新機能追加
- `fix/#42` — Issue #42 のバグ修正
- `refactor/#100` — Issue #100 のリファクタリング
- `chore/#5` — Issue #5 の依存関係更新

**Why:** prefix で変更の性質が一目で分かり、Issue番号でトレーサビリティが確保される。

**How to apply:**
- prefix の判断に迷ったら、ユーザーに確認する
- Issueに紐づかない作業の場合は、Issueを先に作成してから対応する

### 4. 破壊的操作・公開操作は事前確認

以下の操作は**必ずユーザーに確認してから**実行する。一度の許可は当該操作1回限りであり、以後の同種操作も都度確認する。

**破壊的操作:**
- ファイル・ブランチの削除(`rm -rf`、`git branch -D`)
- `git reset --hard`、`git push --force`
- データベース変更、プロセスkill
- コミットの amend・履歴改変

**公開・共有状態への影響:**
- `git push`(特にmain・共有ブランチへ)
- PR・Issueのコメント、Slack・メール送信
- 外部サービスへのデータ送信

**Why:** 取り返しのつかない操作で作業が失われる、または意図しない情報が外部に出るリスクがある。

**How to apply:**
- 操作内容と影響範囲を明示してから確認を求める
- ユーザー指示で明示的に許可された範囲のみ実行する

### 5. コミット・PR運用

**コミット:**
- 意味のある単位で分割する(機能追加とリファクタを混ぜない)
- メッセージは「何を」より「なぜ」を重視する
- フックはスキップしない(`--no-verify` 禁止、明示指示時のみ例外)

**PR:**
- タイトルは70文字以内、簡潔に
- 本文には Summary と Test plan を含める
- 関連Issue番号を本文に記載する(例: `Closes #69`)

**Why:** レビュー容易性・履歴の追跡性を確保する。

## ルールの追加・更新フロー

新たな問題や改善点が出てきたら、以下の手順でこのファイルを更新する。

1. **Issue化**: 「governance」ラベルでIssueを作成し、追加・変更したいルールと理由を記載
2. **ブランチ作成**: `feature/#{Issue番号}` または `chore/#{Issue番号}` で対応
3. **AGENTS.md 更新**: ルール本文・**Why**・**How to apply** をセットで記述
4. **PR作成・レビュー**: 通常のフローに従う

### ルール記述の必須項目

各ルールには以下を必ず記載する:

- **ルール本文** — 何をすべき/すべきでないか
- **Why** — そのルールが必要な理由・背景(将来エッジケース判断に必要)
- **How to apply** — 具体的な手順・適用範囲・例外
