## 絶対ルール

### 1. PR作成・マージ・Issueクローズは明示的指示まで禁止

PR作成・マージ(`gh pr merge`)・Issueクローズ(`gh issue close`)は、ユーザーから明示的に指示があるまで**絶対に実行しない**。

**Why:** ユーザーは変更内容を自分で確認してからマージ・クローズしたい。事前確認なしの実行は信頼を損なう。

**How to apply:**
- コード変更が完了したらブランチをプッシュして止まる
- 「確認いただけましたら指示をお願いします」と明示的に伝える
- 「対応してください」「Issue#XXを対応してください」と言われても、PR・マージ・クローズは別途指示が来るまで実行しない
- **唯一の例外:** ユーザーが「PR・マージ・クローズまでお願いします」と一連の作業として明示的に依頼した場合

### 2. メイン作業は専用ブランチ、派生軽微修正は同居可能 + メイン Issue コメントで記録

mainブランチへの直接コミット・プッシュは禁止。メイン Issue 対応は専用ブランチで進める。

**メイン対応中に発見した軽微な派生修正** (関連バグ・lint 修正・小規模 UI 微調整など) は、原則として **別途 Issue を起票せず** 同じブランチに同居させ、当該メイン Issue のコメントに「派生で発見・対応した内容」として備忘録形式で記録する。

**Why:**
- mainには動作検証済みのコードのみをマージしたい
- 細かい修正ごとに Issue を起こすと Issue 一覧が膨大化し、重要な対応が一覧上で埋もれて検索性が低下する
- 派生修正をメイン Issue のコメントに残すことで、トレーサビリティを保ちつつ Issue 数を抑制できる

**How to apply:**
1. メイン Issue 対応開始前に `git branch -a` で既存ブランチを確認する
2. 対象 Issue 用ブランチが存在する場合、そのブランチが当該 Issue 用かユーザーに必ず確認する
3. ユーザーが「Yes」と答えた場合のみそのブランチに切り替える
4. 既存ブランチがない場合は新規作成する(命名規則は次節)
5. メイン作業中に派生修正を発見した場合の判断基準:
   - **軽微** (1〜2 ファイル相当 / レビュー粒度が独立して読める / アーキテクチャ変更を伴わない) → 同ブランチに同居 + メイン Issue にコメントで「派生で発見・対応した内容」を記録
   - **大規模** (アーキテクチャ変更を伴う / 別レビュアー判断が必要 / 影響範囲がメイン Issue と独立) → 別 Issue + 別ブランチに分割
   - 判断に迷ったら、ユーザーに必ず確認する
6. main へのマージは「ルール 1」に従い、明示的指示があるまで実行しない
7. **例外**: ガバナンスルール変更 (AGENTS.md / `.claude/settings.json` など) は重要度が高く、引き続き専用 Issue + 専用ブランチで分離する (ルール 5 の governance flow を維持)

### 3. ブランチ命名規則

ブランチ名は以下の prefix を使用し、**`{prefix}/#{Issue番号}`** または **`{prefix}/#{Issue番号}-{slug}`** の形式とする。

| prefix | 用途 |
|---|---|
| `feature/` | 新機能追加・機能拡張(ユーザーから見て新しいことができる変更) |
| `fix/` | バグ修正(意図通りに動いていなかったものを直す) |
| `refactor/` | 振る舞いを変えないコード整理・責務分離・命名変更 |
| `chore/` | 上記3つに当てはまらない雑務(依存関係更新、設定変更、CI調整等) |

**例:**
- `feature/#69` — Issue #69 で新機能追加
- `fix/#42` — Issue #42 のバグ修正
- `refactor/#100` — Issue #100 のリファクタリング
- `chore/#5` — Issue #5 の依存関係更新

**サフィックス形式 (1 つの Issue を複数 PR に分割する場合):**

1 つの Issue を複数の段階的な PR に分けて進める場合は、`{prefix}/#{Issue番号}-{slug}` 形式で派生ブランチを切ってよい。slug は段階の内容を端的に表す英小文字+ハイフン (例: `cleanup`, `memo`, `compute`, `preload`, `split`)。

- 例: Issue #107 を 5 段階で進める場合 — `refactor/#107-cleanup`, `refactor/#107-memo`, `refactor/#107-compute`, `feature/#107-preload`, `refactor/#107-split`
- 親ブランチとして `feature/#{Issue番号}` または `refactor/#{Issue番号}` を先に作成しておき、各派生ブランチの PR base にしてもよい(その場合、親ブランチ → main の統合 PR を最後に出す)

**Why:** prefix で変更の性質が一目で分かり、Issue番号でトレーサビリティが確保される。1 Issue が大きくなり段階分割が必要なケースでも、slug で各派生ブランチの粒度を表現できる。

**How to apply:**
- prefix の判断に迷ったら、ユーザーに確認する
- Issueに紐づかない作業の場合は、Issueを先に作成してから対応する
- サフィックス形式を使う場合、slug は分かりやすく統一する(チーム間で命名衝突を避ける)

### 4. 新規ブランチは origin/main を起点にする

新規ブランチを作成する際は、**ローカル main ではなく必ず `origin/main` を起点**にする。

**Why:** 他作業で origin/main が先行している場合、ローカル main は古いままのことがある。古い main から枝を切ると、他Issue・他PRの最新変更が含まれず、デグレに見える状態が発生する(2026-05-01 #77 作業時、#78 マージ済の origin/main を取り込まず、ドロー演出が見当たらず混乱した実例あり)。

**How to apply:**
1. ブランチ作成前に必ず `git fetch origin` を実行する
2. ブランチは `git checkout -b {prefix}/#{Issue番号} origin/main` の形式で作成する
3. ローカル main を最新化してから切る方法でも可: `git checkout main && git pull --ff-only origin main && git checkout -b {prefix}/#{Issue番号}`
4. 既存ブランチに切り替えた場合も、作業開始前に `git fetch && git merge origin/main` で最新を取り込むか、ユーザーに確認する

### 5. 破壊的操作・公開操作は事前確認

以下の操作は**必ずユーザーに確認してから**実行する。一度の許可は当該操作1回限りであり、以後の同種操作も都度確認する。

**破壊的操作:**
- ファイル・ブランチの削除(`rm -rf`、`git branch -D`)
- `git reset --hard`、`git push --force`
- データベース変更、プロセスkill
- コミットの amend・履歴改変

**公開・共有状態への影響:**
- `git push`(特にmain・共有ブランチへ。ただし「ルール6」で定めるIssue専用ブランチへの通常pushは、作業依頼に含まれる確認済み範囲として扱う)
- PR・Issueのコメント、Slack・メール送信
- 外部サービスへのデータ送信

**Why:** 取り返しのつかない操作で作業が失われる、または意図しない情報が外部に出るリスクがある。

**How to apply:**
- 操作内容と影響範囲を明示してから確認を求める
- ユーザー指示で明示的に許可された範囲のみ実行する
- Claude Code の自動承認/拒否ルールは `.claude/settings.json` に集約されている。変更する際は本ファイルのルール追加・更新フローに従い、Issueを立ててPRで反映する
- **「破壊的操作」カテゴリ (`rm -rf`、`git branch -D`、`git reset --hard`、`git push --force` 等) は `.claude/settings.json` で auto-deny を外し allow に置いている。これは「ユーザー明示指示があれば実行可」を意図した設計であり、AI 側の本ルール 5 (= 必ずユーザーに確認してから実行) が唯一の安全網となる。明示指示なしの実行は本ルール違反として扱う**
- 一方、「公開・共有状態への影響」カテゴリ (`git push origin main`、`gh issue close`、`gh pr close`) は引き続き auto-deny を維持する (ルール 1・2 と二重化された安全網)

### 6. Vercelデプロイ確認のため専用ブランチはpushまで行う

Issue対応やコード変更依頼で実装・検証・コミットが完了したら、ユーザーから明示的に「pushしない」「一旦止めて」等の指示がない限り、作業ブランチを origin に push して止まる。

**Why:** 本プロジェクトでは、動作確認を Vercel のプラットフォーム上で生成されるデプロイメントで行う。ブランチが origin に push されていないと、ユーザーがVercel上で実機・実環境に近い確認を進められない。AIエージェントがローカル検証だけで止まると、ユーザー側の確認導線が途切れる。

**How to apply:**
- 実装後、可能な範囲で型チェック・テスト・表示確認を行う
- 意味のある単位でコミットする
- 現在の作業ブランチを `git push origin {branch}` で push する
- push 後は、ブランチ名・コミット・実施した検証を伝えて止まる
- PR作成・マージ・Issueクローズは「ルール1」に従い、明示的指示があるまで実行しない
- main への直接pushは禁止。必ずIssue専用ブランチまたはユーザーが明示した作業ブランチへpushする
- ユーザーが「push不要」「まだpushしないで」等を明示した場合は、その指示を優先する
- push先が共有ブランチである、ブランチ用途が曖昧である、または未確認の変更が混在している場合は、push前にユーザーへ確認する

### 7. コミット・PR・Issue運用

**コミット:**
- 意味のある単位で分割する(機能追加とリファクタを混ぜない)
- コミットメッセージは日本語で記載する
- メッセージは「何を」より「なぜ」を重視し、対応背景・意図・変更内容・原因・対策が第三者にも分かる粒度で書く
- 1行目だけで説明しきれない場合は、本文に背景・原因・対策・検証内容を簡潔に追記する
- フックはスキップしない(`--no-verify` 禁止、明示指示時のみ例外)

**PR:**
- タイトルは70文字以内、簡潔に
- 本文には Summary と Test plan を含める
- 関連Issue番号を本文に記載する(例: `Closes #69`)

**Issue:**
- タイトルは**簡潔・簡素にわかりやすく**(目安: 30文字以内)
  - ❌ 悪い例: 「Claude Code の allowlist を設計し承認プロンプトを削減」
  - ✅ 良い例: 「Claude Code allowlist整備」
- 詳細は本文に書く(背景・目的・スコープ・受け入れ条件)
- 種別ラベル(`governance`, `bug`, `enhancement` 等)を付与する

**Why:** レビュー容易性・履歴の追跡性を確保する。タイトルが長いと一覧で読みづらく、参照(`Issue #XX`)時の文脈把握も遅くなる。詳細は本文で構造化されているべき。

**How to apply:**
- Issueタイトルは「何の話か」が一目で分かる名詞句にする
- 詳細・条件・経緯は本文に書く
- 迷ったら、既存のIssueタイトル(例: #69「AIルール・ガバナンス」)を参考にする

### 8. 重要マイルストーンでの徹底レビュー

以下の3段階では必ず、**Issue #109「共通レビュールール」** に基づき徹底レビューを実施する:

1. **実装計画を策定した直後**(実装着手前)
2. **実装が一通り完了した時点**(コミット・push 前)
3. **動作検証後、最終的にマージする前**(PR レビュー段階)

**Why:**
- 各マイルストーンでレビューを挟むことで、計画段階の見落とし・実装中の品質低下・マージ直前の最終確認漏れを段階的に防ぐ
- レビュー観点を一元管理することで、観点の分散・劣化・更新追従漏れを防ぐ
- AGENTS.md にレビュー観点を転記すると Issue #109 と二重管理になり、片方の更新が片方に反映されないリスクが発生する

**How to apply:**
- レビュー観点の正本は **常に Issue #109 を参照する**(本ファイルに転記しない)。各マイルストーン到達時、`gh issue view 109` 等で **都度最新の #109 内容を取得** してから観点に沿ってレビューする
- レビューはセルフレビュー(AI 自身)と必要に応じてユーザーレビューの両軸で実施する
- 指摘事項・改善点が全て解消されるまで次の段階に進まない
- レビュー結果は当該 PR・Issue のコメント、または会話上に明示的に残す(「観点 X について確認: ◯◯ で問題なし」等)
- Issue #109 の観点が更新された場合、本ルールには影響を与えず、参照先(#109)のみが更新される(単一情報源)

### 9. ローカル作業スペースは原則 Worktree を使用 + 完了後にブランチ・Worktree を削除

複数 Issue を並行進行することが多い本リポジトリでは、メインの作業ディレクトリでブランチ切替を続けると、別件対応中の未コミット変更や別ブランチとの競合 (意図せぬ巻き込み・取り違え) が発生しやすい。また、対応完了 (PR マージ + Issue クローズ) 後の作業ブランチ・Worktree を残し続けると、作業ディレクトリ内に古いブランチが滞留し、現在進行中の作業との混同・参照ミス・ディスク肥大を招く。

**Why:**
- 並行作業中の別ブランチとの競合を避け、各 Issue の作業を独立した状態に保ちたい
- 完了済みブランチ・Worktree が残ると、状態管理が複雑化し、誤って古いブランチを参照・編集する事故の温床になる

**How to apply:**

*作業開始時:*
- 新規対応を始める前に、**Worktree (`git worktree add`) で作業ディレクトリを切り出すか、メインディレクトリでブランチを切替えて進めるかをユーザーに必ず確認する**
- 既に別件対応中のブランチや未コミット変更がメインに残っている場合は、Worktree を強く推奨する
- ユーザーが明示的に「Worktree で」「メインで」等の指示を出した場合はそれに従う
- Worktree 配置先は `.claude/worktrees/issue-{番号}` を既定とする(既存運用との整合)

*対応完了時 (= PR マージ + Issue クローズが完了したもの):*
- 作業に使った **Worktree を削除する** (`git worktree remove <path>`)
- 該当 **ローカルブランチを削除する** (`git branch -d <branch>`)
- **origin の該当ブランチも削除する** (`git push origin --delete <branch>`)
- 削除はルール 5「破壊的操作・公開操作は事前確認」の対象なので、削除対象一覧 (Worktree path・ローカルブランチ名・origin ブランチ名) をユーザーに提示し、確認を取ったうえで実行する

## カード将棋: 新規カード追加時のチェックリスト

カード将棋 (card-shogi variant) で新しいカードを `CARD_DEFS` に追加するとき、以下を確認すること。共通基盤の挙動 (待った・DB 保存・AI ガード等) は基本的に**ヘルパで自動的に正しく動く**ようになっているが、新しい横断的概念を導入するときは更新が必要。

**Why:** 過去に「二手指し (#82) 追加時に `待った` 制約が破綻」「同種類の event 追加で複数箇所に判定ロジックが分散しデグレ」など、共通基盤のクロスカット制約を考慮し忘れたバグが繰り返し発生したため。

**How to apply (実装者向け):**

1. **カード定義** (`src/lib/shogi/cards/definitions.ts`)
   - `CardDefinition` の必須フィールドをすべて埋める
   - `status` は新規実装中なら `"preparing"`、検証用見本なら `"draft"`、本実装完了後は `"active"`
   - `useConditionDescription` は `CARD_USE_CONDITIONS` に登録した条件と整合させる
   - **`checkUsage` (王手中の使用可否) を必ず明示** (Issue #82)。判断指針:
     - **大前提**: 自分の手番開始時、王手中なら必ず1手で回避できる手が存在する (詰みなら手番が回らない)
     - `"forbidden"`: そのカード効果が王手回避になり得ない (盤上駒退避系・盤面に作用しないカード・トラップ)
     - `"conditional"`: 通常の1手の一部パターンでのみ王手回避になる (合駒系)。target ありなら動的判定が動く
     - `"unconditional"`: 通常の1手分以上の選択肢を提供する (= 必ず1手回避手を取れる。例: 二手指し)
     - トラップカードは原則 `"forbidden"`
     - 詳細: Issue #82 のコメント「王手時カード使用可否の検討観点」を参照

2. **効果適用関数** (`src/lib/shogi/cards/effects.ts`)
   - target ありカードは `applyXxx` を実装し、`simulateCardEffect` の switch にも追加
   - `isValidCardTargetSquare` で対象マスの妥当性判定を実装 (王手中の使用条件含む)
   - target なしカード (盤面を変えないもの) はデフォルト動作で OK

3. **reducer 効果分岐** (`src/hooks/card-shogi/reducer.ts` の `CONFIRM_PLAY_CARD`)
   - `def.effectId === "..."` の分岐に新カード処理を追加
   - 1 ターンに複数 ply 消費するカード (二手指し系) は `state.doubleMove` パターンを参考に

4. **横断制約の確認** (このチェックリストの本丸)
   - **新たな event kind を追加する場合**:
     - `src/lib/shogi/cards/types.ts` の `GameEvent` ユニオンに追加
     - **`src/hooks/card-shogi/undo-policy.ts` の `isCardOpEvent` を更新**
       (待った可否判定で漏れなく block されるように)
   - **複数 ply のカード (= 自分のターンで 2 つ以上 `moveEvent` を発行する) を追加する場合**:
     - `getUndoScope` (undo-policy.ts) は同色連続を 1 ターン扱いで自動対応
     - `state.doubleMove` のような明示的な「ターン継続中」フラグを reducer に持つこと
     - DB 保存スキップ (`src/hooks/use-card-shogi-game.ts` の save useEffect) も該当フラグを考慮

5. **テスト**
   - `npm run test:ci -- src/hooks/card-shogi/__tests__/undo-policy.test.ts` が緑であることを確認
   - 新カードの効果関数のユニットテストを `effects.test.ts` に追加
   - 1 ターン複数 ply 系の場合は reducer 統合テストも追加

6. **prisma seed**
   - `ALL_CARD_DEFS` 経由で自動的に Card マスタ・DeckEntry・PlayerCardCollection に投入される
   - 既存ローカル DB に反映するには `npm run db:seed` を実行

## ルールの追加・更新フロー

新たな問題や改善点が出てきたら、以下の手順でこのファイルを更新する。

1. **Issue化**: 「governance」ラベルでIssueを作成し、追加・変更したいルールと理由を記載
2. **ブランチ作成**: `feature/#{Issue番号}` または `chore/#{Issue番号}` で対応
3. **AGENTS.md 更新**: ルール本文・**Why**・**How to apply** をセットで記述
4. **PR作成・レビュー**: 通常のフローに従う

### ルール記述の必須項目

各ルールには以下を必ず記載する:

- **ルール本文** — 何をすべき/すべきでないか
- **Why** — そのルールが必要な理由・背景(将来エッジケース判断に必要)
- **How to apply** — 具体的な手順・適用範囲・例外
